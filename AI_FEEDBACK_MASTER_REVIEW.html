BUG 1: Kelly Fraction Mismatch (Frontend vs Backend)

Impact: Frontend sizes positions at exactly 2x the backend's Kelly fractions. A trade auto-executed from the dashboard will be double the intended risk vs the same trade executed by the 24/7 backend scanner. These appear to be leftover Quarter-Kelly values that were never updated to Eighth-Kelly.
Research verdict: Eighth-Kelly fractions (backend values) are correct for the current stage. Academic sources confirm Kelly/8 is standard for systems with <55% WR and <1000 trades. Max expected drawdown at Eighth-Kelly: ~6.25%.

BUG 2: sqrt(n) Correlation Discount is Mathematically Wrong

Current code: when 3+ assets signal same direction, each position reduced by 1/sqrt(n).

The math: 4 correlated positions at 50% size each = 200% total exposure. You've doubled correlated exposure while thinking you reduced it. For crypto (BTC-ETH rho ~0.85, BTC-SOL rho ~0.72), sqrt(n) massively overstates diversification benefit.
Correct formula: sigma_p = sigma * sqrt(1/n + (1 - 1/n) * rho)
Or simpler: use targetHeat / n allocation. If max 3% heat on correlated group and 4 positions, each gets 0.75%.

Source: Standard portfolio theory. Confirmed by AnalystPrep, Stanford, portfolio risk literature.

BUG 3: Kill Switch Auto-Dismiss on Confirm Mode

In "confirm" auto-trade mode, the notification popup auto-dismisses after 30 seconds. If a kill-switch event fires (20% drawdown, 5 consecutive losses), this popup also auto-dismisses.

Research verdict: TRUE safety hole. FIA best practices state "an automated trader should not be able to override a kill switch invoked by the broker." PMC documented real cases where kill switch activation was delayed 30+ minutes with catastrophic results. Kill switch events must require active user acknowledgment.
